# 2026-05-10 Phase 5D 测试日志

## 目标

扩大 LLM shadow eval 样本，基于 labeled mock emails 分析 mismatch，并通过 prompt/taxonomy 优化建立可复盘的 LLM 调试闭环。

## 当前约束

- 只使用 mock 邮件。
- 不连接真实邮箱。
- 不修改邮箱状态。
- 不让 LLM 执行写工具。
- 不记录 API key、真实邮箱内容或大段完整 rows。

## Mismatch 分析

当前 mismatch：

- `email-035`: `finance` invoice 被预测为 `action_required`。类型：taxonomy confusion。原因：邮件包含 due tomorrow / service interruption，模型优先捕捉到了行动要求，但产品 taxonomy 里 finance 优先级应高于 action_required。
- `email-032`: meeting reminder 被预测为 `high` / `pay_attention`。类型：importance/action calibration。原因：模型把“10 minutes”理解成高紧急程度，但产品定义里普通会议提醒应保持 `medium` / `schedule`。
- `email-030`: operational alert 出现 API timeout。类型：provider stability / schema unavailable。原因：真实 LLM API 调用超时，不能归因于 prompt 分类错误。
- `email-028`: finance receipt 被预测为 `medium` / `archive`。类型：importance/action calibration。原因：模型把 receipt 视作自动收据，低估了财务类邮件默认需要 review 的产品定义。

## Prompt 优化

待执行。优先做 prompt-only 修改：

- 明确 category priority。
- 明确 finance 即使需要处理，也保持 `finance` category。
- 明确 receipt / invoice / billing / renewal 默认 `finance` + `review`，importance 至少 `medium`，涉及续费/金额/服务风险时为 `high`。
- 明确普通 meeting reminder 通常是 `meeting` + `medium` + `schedule`，不要因为时间近就自动升级为 `high` / `pay_attention`。
- 明确 social notifications 默认 `noise`。
- 保留 JSON-only 和枚举校验要求。

- 5 条 `APITimeoutError: Request timed out.`

结论：

- 36 条串行 LLM eval 会暴露 provider timeout 问题。
- timeout 会把重要邮件变成 false negative，因此必须把 provider failure 和 semantic mismatch 分开分析。
- 后续给 LLM shadow eval 增加可配置 `timeout` / `max_retries`，默认轻量 retry。

## 结论

Phase 5D 达到阶段目标：

- prompt/taxonomy 优化显著提升 LLM shadow eval。
- `important_recall`、`important_precision`、`noise_filter_precision` 在 36 条完整样本上达到 1.0。
- provider timeout 被识别为独立工程风险，并通过 `timeout` / `max_retries` 缓解。
- 剩余 mismatch 都不影响是否汇报重要邮件，属于 taxonomy/action/importance 边界，可在后续产品定义中继续细化。

面试时可以强调：

> 我没有只看一次 LLM 输出，而是用 labeled mock dataset 做 shadow eval。先记录 baseline，再按 mismatch 类型优化 prompt/taxonomy，并把 provider timeout 和分类错误分开处理。最后保留 rule baseline 作为稳定参照，LLM 输出仍然需要 JSON 解析、枚举校验和权限边界保护。
