# Phase 7B Server Hardening

日期：2026-05-10

## 目标

在接入真实 Outlook read-only provider 前，先收紧 server 安全边界和运行时状态一致性。

## 已完成

- API token：`WISPERA_AUTH_TOKEN` 设置后保护除 `/health` 外的接口。
- 开发工具：`list_files`、`read_text_file`、`run_shell_command` 默认不注册，只在 `WISPERA_DEV_TOOLS=1` 时启用。
- 工具策略：敏感路径拒绝读取，shell 控制符、重定向、管道和高风险命令被拒绝。
- 脱敏：trace 和 pending approval 只暴露参数摘要，不保存完整草稿正文。
- 评估边界：`email_eval_report` 只能写入 `docs/test-logs/`，mock eval 固定使用 `MockEmailProvider`。
- Provider factory：`WISPERA_EMAIL_PROVIDER` 当前只接受空值或 `mock`，未知 provider 直接失败。
- 状态锁：`MemoryStore` 和 `ToolRegistry` 使用进程内 `RLock`。
- Scheduler 去重：notification 创建收敛到 `create_email_notification_once()`；SQLite 模式下同一事务插入 reported id 并保存 notification，跨两个 store 共享 DB 时也能避免重复通知。
